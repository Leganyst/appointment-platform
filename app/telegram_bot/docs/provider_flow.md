# Provider Flow (Context Note)

## 0) Старт / идентификация
- `/start`: приветствие, авто-ID по Telegram ID, пользователь создан/найден, упоминание управления расписанием, кнопки «Выбор роли» / «Главное меню».

## 1) Настройка профиля представителя
- Выбор роли «представитель услуг» кнопкой.
- Запрос доп. сведений: наименование услуги, описание, контактный телефон/канал.
- Подтверждение → возврат в главное меню.

## 2) Главное меню представителя
- Доступ к: поиск услуг/представителей, просмотр своих записей, управление расписанием (для представителя), профиль/справка. Требование: управление расписанием и просмотр записей доступны из меню.

## 3) Экран управления расписанием
- Переход из главного меню.
- Показывает список слотов (доступные/занятые), краткую инфо о записях клиентов на слоты, кнопку возврата в меню.

## 4) Создание слота
- Кнопка «Добавить слот» → ввод даты/времени/длительности.
- После сохранения сервисное сообщение об успехе/ошибке.
- Повторяющиеся события допускаются (из ТЗ), форма не детализирована.

## 5) Изменение слота
- Выбор слота → «Изменить».
- Разрешено менять дату/время/длительность только если слот не занят.
- После изменения сервисное сообщение.

## 6) Удаление слота
- Выбор слота → «Удалить».
- Допустимо, если не нарушает целостность данных о бронированиях.
- После удаления сервисное сообщение.

## 7) Работа с записями клиентов
- На экране управления расписанием видно краткую инфо о записях клиентов на слоты.
- Подтверждение записей выполняет представитель через админ-интерфейс (бот).
- Отмена записи/слота инициируется представителем, пользователь получает сервисные сообщения с результатом и дальнейшими действиями.

## 8) Профиль и помощь (представитель)
- Отображает: имя в Telegram, роль, контакты; описание системы; перечень команд/разделов (в т.ч. «Управление расписанием»); кнопку возврата в меню.

## Примечание
- Требования к инлайн-кнопкам/структурированным спискам и сервисным сообщениям остаются общими (как в клиентском флоу): понятные статусы и предложения дальнейших действий.
